# 作业报告：详细提示词 → SVG 徽标生成（LoRA 微调）

## 1. 任务概述

这次作业拿到的是一个配对数据集（train 219 条，valid 17 条），每条包含一个详细视觉描述和一个 Sonnet 画的 SVG 徽标。要做两件事：写一个程序化的奖励函数来评估生成的 SVG 好不好，以及用 LoRA 把 Gemma 3 270M 这个小模型微调到能根据提示词画 SVG。

## 2. 奖励函数

奖励函数分了四个维度，每个维度都对应实际代码中可检查的项目，不依赖任何外部模型或人工判断。

有效性检查占了最大的权重（35%），关注 SVG 能不能被正确渲染。首先看是不是合法 XML，然后检查根节点是不是 `<svg>`、有没有 `xmlns`、`viewBox` 是不是 `0 0 256 256`。我还额外加了一个 path 语法校验——XML parser 能过不代表渲染器能过，比如 `C 100 100 108 108 128` 这条 cubic bezier 命令只给了 5 个数值但需要 6 个，XML parser 不报错但浏览器直接拒绝。这个是我 debug 的时候发现的，之前有几条生成的 SVG 明明 XML 解析过了但图片就是显示不出来，追了半天发现是 path 数据的问题。

结构维度（25%）管的是 SVG 是否"正常"。坐标是不是在 0-256 这个画布范围内、用的颜色是不是 2 到 8 种、元素数量有没有离谱（太少说明空洞，太多说明在堆砌）。颜色和元素数量的阈值是靠观察训练数据定的，Sonnet 画的 SVGs 基本都在这个范围里。

内容相关性（20%）想看的是生成的 SVG 和提示词之间有没有关系。我从提示词里提取颜色词和形状词，然后在 SVG 里搜。虽然这样很粗糙——搜到 "circle" 不代表真的画了个圆——但作为一个自动化的 proxy 够用了。

退化检测（20%）是专门抓模型崩掉的各种花样的，大部分来自我对训练初期 bad case 的观察：输出短到没有内容、全是 markdown 代码块而不是裸 SVG、同一行文字重复了几十次、甚至输出过印地语和马来语。

最后用 Sonnet 画的 17 条验证集跑了一下，平均 0.835，全部 17 条都在 0.80 以上。说明函数对高质量 SVG 的识别是准的，同时不给满分也留了点区分空间。

## 3. 训练方法与踩坑

模型选了 Gemma 3 270M Instruct，从 ModelScope 下的。LoRA 加在 q_proj、k_proj、v_proj、o_proj 和 FFN 的 up/down/gate 上，rank 16，总共 3.8M 可训参数。训练 15 个 epoch，lr 1.5e-4，有效 batch size 8，max_length 8192 确保最长的样本也不会被截。loss 只算 assistant 那部分，prompt 部分全部 mask 成 -100。

这里踩了不少坑，最大的一个是一开始下了 base 模型。训练完发现模型输出的是 system prompt 的重复文本，完全不生成 SVG。base 模型不理解 `<start_of_turn>` 这种对话格式，SFT 教不了。换成 instruct 版本以后才开始正常学。

另一个坑藏得比较深。样本截断的时候 full_ids 被截掉了一部分头部 token，但 prompt_ids 没有同步截。这导致 labels 里 prompt 遮罩错位——本应该标成 -100 的部分没标，应该学的那段 SVG 代码反而被标成了 -100。还有一个类似的问题在 eval 生成的时候，`tokenizer.decode` 加了 `skip_special_tokens=True` 会把 chat template 的特殊 token 删掉，后续用字符串长度切片就全错了——生成的 SVG 文件里存的其实是 system prompt 的文本。修成基于 token ID 的切片就好了。

推理速度也很折磨，270M 模型在 A100 上也就能跑 33 tok/s 左右。一个样本要两分钟，17 条加上基座对比就是快一小时才出结果。没办法，模型太小了 GPU 根本吃不饱，大部分时间花在了 kernel launch 开销上。

## 4. 实验结果

在 valid.jsonl 全部 17 条上跑的结果摊开来看比较直观。

没训过的 instruct 模型基本不会画 SVG。它知道要"回应"但技能为零——输出全是 markdown 代码块裹着的碎片，带有 `stroke-dash=0` 这种不存在的属性，甚至有好几条末尾是印地语字符和 "dalam dalam dalam" 的重复。平均 reward 0.048，17 条里零条能用。

训完 15 个 epoch 之后改善很明显。平均 reward 到了 0.475，超过一半（9/17）的样本能生成合法的 SVG。做得好的那些其实已经很接近 Sonnet 了——高分组里中位数 reward 是 0.778，跟 Sonnet 的 0.833 差得不多。这些样本的 SVG 结构完整，标签都对，颜色用的是提示词里指定的值，gradient 和几何元素组织得也有样子。

但剩下 8 条（47%）是全挂的，而且挂法全部一样：SVG 没闭合。模型吭哧吭哧写了几千个 token 的 SVG 代码，到了 3072 token 的上限还没写到 `</svg>`。这 8 条不算格式错误——开头和中间的结构都对——纯粹是内容太膨胀了，写到一半没写完。这 8 条恰好是对应着特别复杂、提示词很长的那些徽标描述。

一个有意思的点是分数分布是两极化的——要么 0.8 要么 0.1，中间没有过渡。这说明模型的失败模式不是"慢慢变差"，而是"要么对要么就写不完"。改善的方向也比较清楚：加个 rep_penalty 抑制内容膨胀，或者提高 max_new_tokens 的冗余。

## 5. 总结

做了这些事情之后有几个比较明确的结论。第一，程序化 reward 函数虽然粗糙但是有效的，四个维度确实能区分出好和坏的 SVG，而且 path 语法校验补上了 XML parser 的一个盲区。第二，LoRA 微调带来了实际提升，基座模型几乎什么都不会，训完有一半的机会画出合格的 SVG，而且成功的时候质量不差。第三，最大的问题是剩下的 47% 失败率，但这看起来是可以修的——根因很单一（生成内容膨胀导致截断），不是那种到处漏水的系统性失败。

如果要继续做的话，加个 `repetition_penalty`、多做几个 epoch、或者换大一点的模型，都是现成的方向。当前 state 已经把从 0 到能画这件事做通了，剩下的是提高通过率的问题。
